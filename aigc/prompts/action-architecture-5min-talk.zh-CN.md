# Action 架构评审 5 分钟口播提纲

> 说明：此提纲用于开源项目讨论场景，结论基于当前仓库可见实现（截至 2026-02-28）。

## 0:00 - 0:40 开场

今天评审的核心不是“重构与否”，而是确认 mss-boot / mss-boot-admin 的 Action 架构是否还能支撑下一阶段增长，并给出低风险演进路径。

一句话结论：**优先保持主架构连续性，并优先补齐治理能力。**

## 0:40 - 1:40 现状与价值

- 我们当前以 `Action + Controller + Provider` 为核心，形成统一资源语义。
- 控制器大多数保持轻量，CRUD 开发效率高，接口风格一致。
- 同一动作语义可以落到 GORM / MGM / K8S，不同后端不需要重写控制器。

这套架构已经在 admin 项目中证明了规模化可复用能力。

## 1:40 - 2:40 主要问题

- 调试链路偏长：Router -> Controller -> Action -> Hook -> DB。
- `Control` 在复杂场景里容易承载过多分支，维护成本上升。
- Hook 增长后，执行顺序和副作用管理难度增加。

也就是说，当前瓶颈不是功能不足，而是治理成本在上升。

## 2:40 - 3:50 评审结论

三点建议结论：

1. 优先保留 Action 架构主线，避免高风险重写。
2. 本季度优先投入可观测与治理能力，而不是新增高风险架构实验。
3. 对高复杂模块做“渐进式动作拆分”，降低 `Control` 负担。

## 3:50 - 4:40 3 个月计划

- **M1 可观测性基线**：打通 Action 链路日志、Hook 耗时、错误率与 P95/P99。
- **M2 复杂度治理**：新模块默认 Create/Update 分离；高频模块降低 `Control` 复杂度。
- **M3 Hook 规范化**：统一 Hook 顺序、错误语义、幂等规则与测试模板。

## 4:40 - 5:00 收尾与决策请求

建议本次会议讨论并确认三项决策：

1. 架构主线不变。
2. 资源优先投入治理能力建设。
3. 每两周做一次 Action 健康度复盘（错误率、耗时、复杂度）。
