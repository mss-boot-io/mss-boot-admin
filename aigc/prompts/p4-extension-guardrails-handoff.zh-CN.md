## 1. P4 定义回顾

### 目标

> 保留平台的扩展能力，但防止扩展绕开治理与运营主线。

### 覆盖范围

- 国际化
- 对象存储/上传
- WebSocket 事件能力
- API-first 扩展方式

### 核心问题

1. 新能力如何接入现有治理与运营体系？
2. 扩展模块应遵守什么边界？
3. 哪些扩展属于平台能力，哪些属于业务层能力？

### 预期结果

1. 扩展能力继续保留，但不会重新把产品叙事带回"功能散点化"
2. 新能力接入更容易被评审、被验证、被文档化

### 2.3 核心问题回答

**Q1: 新能力如何接入现有治理与运营体系？**

| 治理要求 | 接入方式 |
|----------|----------|
| JWT 认证 | 使用 `middleware.Auth.MiddlewareFunc()` 或 `controller.WithAuth(true)` |
| Casbin 权限 | 在 Casbin 策略中定义 API 权限，菜单管理中绑定 |
| 数据权限 | 使用 `controller.WithScope()` 实现数据隔离 |
| 租户隔离 | 继承 `ModelGormTenant` 模型基类 |

**Q2: 扩展模块应遵守什么边界？**

| 能力 | 边界规则 |
|------|----------|
| 国际化 | ISO 639-1 语言代码，静态翻译在前端，动态翻译在后端 |
| 存储 | 必须使用 `{userID}/` 路径隔离，禁止直接文件系统访问 |
| WebSocket | 使用预定义事件类型，消息大小 ≤512KB，心跳超时 5 分钟 |
| API-first | 通过 `AppendController` 注册，实现标准接口 |

**Q3: 哪些扩展属于平台能力，哪些属于业务层能力？**

| 层级 | 能力类型 | 示例 |
|------|----------|------|
| 平台能力 | 框架提供的通用机制 | Controller.Simple、Hook、WebSocket Hub |
| 业务能力 | 基于平台机制的业务实现 | 具体业务模型、自定义事件处理 |
