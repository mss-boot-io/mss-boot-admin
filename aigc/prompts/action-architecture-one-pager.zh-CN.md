# Action 架构评审一页版（mss-boot / mss-boot-admin）

> 说明：本页基于当前仓库可见实现整理（截至 2026-02-28），用于开源协作评审与路线讨论。

## 一、10 条关键结论

1. 当前框架核心竞争力是 **Action 驱动的资源编排**，极大减少 CRUD 重复开发。
2. `Controller -> Action -> Provider` 分层清晰，能同时支持 GORM / MGM / K8S。
3. 业务控制器普遍较薄，利于团队快速扩展资源接口。
4. Hook + Scope 提供了多租户、鉴权、审计等横切能力的统一注入点。
5. 路由自动装配机制保证了接口风格一致性，降低前后端协作成本。
6. 主要风险是调用链较长，定位问题需要跨层排查。
7. `Control` 同时承载创建与更新，在复杂业务下可读性和可维护性下降。
8. Hook 数量上升后，执行顺序与副作用管理成本显著增加。
9. 虚拟模型等运行时机制提升灵活性，同时带来编译期安全性不足。
10. 该架构适合中后台平台化建设，但需要配套可观测与治理规范。

## 二、当前判断

- **总体评价**：架构方向正确，工程化程度高，适合持续演进。
- **优先问题**：不是“要不要重构”，而是“如何提升可观测性和可治理性”。
- **策略建议**：优先保留 Action 主体设计并逐步治理复杂度，是否重构按阶段数据再决策。

## 三、3 个月行动项（按优先级）

## M1（第 1 个月）：可观测性基线

目标：让每条 Action 链路“可看见、可定位、可度量”。

- 给 Action 统一输出结构化日志字段：controller、action、provider、model、tenant。
- 给 Hook 增加开始/结束/耗时日志。
- 增加核心指标：请求量、错误率、P95/P99（按 action 维度）。
- 产出《Action 排障手册 v1》。

验收标准：线上任一 Action 500 错误，5 分钟内定位到具体层与具体 Hook。

## M2（第 2 个月）：复杂度治理

目标：控制 `Control` 复杂度，降低维护成本。

- 新增模块默认拆分 Create/Update，减少 `Control` 过载。
- 对高频变更模块设复杂度阈值（函数行数、分支数）。
- 建立评审规则：超过阈值必须拆分动作或下沉服务层。

验收标准：Top 10 高频模块中，至少 60% 完成 `Control` 复杂度下降。

## M3（第 3 个月）：Hook 治理规范

目标：把“可插拔”变为“可治理”。

- 定义 Hook 分层与顺序规范（pre-auth / pre-db / post-db / post-response）。
- 统一 Hook 错误语义（可重试、不可重试、告警等级）。
- 形成 Hook 模板（含幂等说明、回滚策略、测试样例）。

验收标准：新增 Hook 全部通过规范检查，并具备最小回归测试。

## 四、评审会决策建议

- 决策 1：优先以 Action 架构为主线推进，是否大规模重写基于后续指标评估。
- 决策 2：本季度聚焦“治理能力”，不新增高风险架构试验。
- 决策 3：设立每两周一次的 Action 健康度复盘（错误率、耗时、复杂度）。

## 五、可直接复用会议结论

建议结论：优先保留 Action 架构并持续演进。短期补齐可观测与治理规范，中期推进动作拆分与类型化增强，以低风险方式提升稳定性与可维护性。
